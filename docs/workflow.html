ERCOT: Weather → Net Load → Gas Burn → Price

Daily trading workflow for the ERCOT weather‑driven fundamentals dashboard, plus a figure‑by‑figure reference (input · output · how to read) for every panel on the site. Each panel is built and runs today. NG = natural gas.

The flow — data to signal

1 · Sources

Where the data comes from

  • StormVista built — grid‑corrected temperature (city‑extraction) + load‑weighted degree days (CDD/HDD) + GEFS ensemble (paid)
  • Meteologica built — demand (system + by zone), wind, solar, net‑load band, battery (forecast + obs + ECMWF‑ENS)
  • Open‑Meteo built — ERA5 archive for the year‑over‑year lookback views (free)
  • EIA‑930 built — actual hourly ERCOT gas generation (free key, ~13 h lag)
2 · Processing

What we compute

  • net load = demand − wind − solar
  • implied burn = (net load − baseload) × heat rate → Bcf/d; actual burn = EIA gas × HR
  • anomaly = forecast − 30‑yr normal; ensemble band = p10/p90 across members
  • weather→load fit a + b·DD, validated out‑of‑sample (leave‑one‑out CV)
  • weather‑normalization (hold temperature constant) to expose solar/battery drift
3 · Output

Two dashboards

  • 📡 Live monitor — forward‑looking, 1–16 days
  • 🔋 Weather‑normalized history — structural drift, positioning

streamlit run app/app.py

Figure‑by‑figure reference — input · output · how to read

📡 Live monitor — forward‑looking

① Temperature anomaly map StormVista Open‑Meteo

Input
Forecast views (High °, Low °, Anomaly): StormVista GFS city‑extraction/corrected‑max‑min → per‑station daily high/low (°F); each county takes its nearest station; anomaly = forecast high − the 30‑yr ERA5 normal. Lookback views (vs Yesterday … vs 1 yr, vs 10‑yr normal): Open‑Meteo — now = GFS Seamless, reference = ERA5 archive.
Output
Per‑county Texas choropleth + per‑zone (county‑mean) table. A Forecast‑day slider (1–16) on the forecast views; red = warmer, blue = cooler.
Read: a big +anomaly over a load‑heavy zone in summer (Coast/Houston, North‑Central/DFW) ⇒ AC surge ⇒ flag for burn. Winter −anomaly ⇒ heating + freeze‑off risk. The vs‑normal view is the demand surprise the market may not have priced. Caveat: station‑resolution — neighbouring counties share one station.

🗺️ ERCOT forecast demand by zone Meteologica aggregate

Input
Meteologica's published hourly demand forecast (MW) for each of the 8 ERCOT weather zones (already weather‑driven by their internal model). PowerDemand/Forecast/<zone>/Total/Hourly
Compute
Pick a forecast day → that day's peak (max) or mean → ÷1000 → GW. No model on our side — pure slice + aggregate.
Output
Texas choropleth, each zone block coloured/labelled by GW + a forecast‑days × zone heatmap table (whole horizon at once). Forecast‑day slider + Peak/Mean toggle.
Read: North‑Central (DFW ~25 GW) + Coast (Houston ~21 GW) dominate; total ≈ ERCOT peak. Slide the day to watch it evolve. Pairs with ①: map = where it's hot, this = where that heat becomes load.

③ ERCOT degree days — demand‑weather StormVista band + anomaly

Input
StormVista population‑weighted ERCOT WDD — four files: deterministic GFS forecast, GEFS members, recent actuals, 30‑yr normal. CDD = max(T̄−65,0) cooling; HDD = max(65−T̄,0) heating.
Compute
p10/p90 band = quantiles across members; anomaly = forecast − normal (subtraction). Degree days themselves are pre‑computed by StormVista.
Output
Forecast CDD/HDD line + GEFS p10–p90 band + dashed 30‑yr normal + green recent actuals; metrics (next‑day level, anomaly, ensemble spread); look‑ahead slider.
Read: the gap to the dashed normal is the anomaly — above normal ⇒ more demand ⇒ bullish power/gas burn; a wide band ⇒ low confidence, size down. CDD = summer AC, HDD = winter heating (toggle to the in‑season one). A pure‑weather signal — no growth/renewables baked in.

③·a Weather → load StormVista Meteologica OLS

Input
X = StormVista load‑weighted CDD/HDD or absolute high °F (toggle). Y = Meteologica daily‑mean demand & net load (GW). One point = one forecast day in the look‑ahead window.
Compute
Daily‑mean the load (drop <24 h days), inner‑join by date, fit each series by OLS: slope s = cov(y,x)/var(x) → GW per degree.
Output
Scatter: two point‑clouds + two dotted fit lines, each labelled with its slope (e.g. demand +0.85 GW/CDD, net load +0.12).
Read: demand tracks weather cleanly (steep +slope); net load is decoupled (flat slope) because wind+solar swing it — the vertical gap between the clouds is renewable generation. A hot day ⇒ high burn only if it's also calm/cloudy. Watch degree days and renewables together.

③·b Hourly / sub‑daily StormVista Meteologica

Input
StormVista 3‑hourly station temperature (°F) — load‑weighted ERCOT average or a single city (Houston/Dallas/…); Meteologica hourly net load (GW, whole system).
Output
Dual‑axis chart: temperature (left °F, red solid) vs net load (right GW, blue dotted). A Focus‑day selector zooms to one forecast day's intraday shape; station multiselect.
Read: temperature peaks mid‑afternoon; net load ramps into the evening as solar fades — that evening window is where peaker burn and price spikes live. Pick Houston/Dallas to see a city's own swing.

ERCOT net load forecast (demand − wind − solar) Meteologica net load

Input
Meteologica system demand, wind, PV (+ ECMWF‑ENS members). net load = demand − wind − solar.
Output
Net‑load line + ECMWF‑ENS p10–p90 fan + p50; metrics (current/peak net load, peak renewables, peak p50).
Read: high peak + steep evening ramp ⇒ tight system ⇒ ancillary / peak‑price risk. A wide, high P90 ⇒ spike risk ⇒ long power / long volatility (convex payoff). This sets the marginal unit and price.

Implied power‑sector gas burn Meteologica EIA‑930 burn proxy

Input
Forecast burn = (net load − must‑run baseload) × heat rate → Bcf/d (sliders for HR & baseload); actual burn = EIA‑930 ERCOT gas generation × heat rate (measured, no baseload). + ENS band.
Output
Forecast burn (brown) + actual EIA burn (green) + ENS band + max‑ramp metric.
Read: forecast burn > market consensus ⇒ long NG & power; < ⇒ short. Green persistently above brown ⇒ your proxy is low (raise baseload/HR, or trust EIA). The calibration check on the whole chain.

Demand decomposition (net load + wind + solar = demand) Meteologica

Input
Meteologica net load, wind, solar — the additive stack that sums to total demand.
Output
Stacked area: net load + wind + solar.
Read: is high net load from high demand, or from low renewables? High renewable share ⇒ less burn (bearish NG); low wind/solar on a hot day ⇒ net load & burn spike (bullish). Tells you why net load is where it is.

🔋 Weather‑normalized history — positioning

Temperature box (per period) Meteologica ERA5

Input
Daily temperature for each selected period (years or months).
Output
Box plot — median / quartiles / mean ◇ per period.
Read: confirms the periods you're comparing actually had different weather — the confound the curves below strip out, leaving only structure.

① Weather → demand: correlation & drift Meteologica degree days fit

Input
Daily demand or net load vs CDD/HDD, per period (years/months). Fit one line a + b·DD per period.
Output
Scatter (dots = binned means, dotted = fit) + coefficient table: r, baseline a, slope b (GW/°day), value @DD=15.
Read: r ≈ 0.9 ⇒ weather almost fully explains demand. The @DD=15 column = demand at identical weather; its climb (53.5 → 61.8 GW, 2022→25) is pure structural growth (data centres / electrification), not weather. Recalibrate seasonal burn to today's level or you'll be structurally too short NG.

🎯 Validation — backtest the model out‑of‑sample CV

Input
Pooled daily demand vs DD; a resolution toggle (Season / Month / Week / Daily) + optional weekday/weekend split.
Compute
Leave‑one‑out CV (numpy least‑squares) — every point predicted out‑of‑sample so a finer resolution can't win by memorising.
Output
OOS MAE / bias / R², overfit gap, weekend effect, the train/test sizes, and a predicted‑vs‑actual scatter.
Read: OOS MAE ≈ 1 GW (~1.5%) at season/month ⇒ trust the demand/burn forecast and size accordingly; Week overfits, Daily can't fit (flagged honestly). The negative bias = the drift — a stale model under‑predicts. The weekend term only helps because it survives OOS.

② Daily shape at a held temperature (diurnal) Meteologica

Input
Hourly load by hour‑of‑day, within a held temperature bin, per period.
Output
24‑hour load‑shape curve per period.
Read: at the same weather, midday dip = solar, lower evening peak = batteries. A lower 2025 evening peak vs 2022 ⇒ batteries shaving the peak ⇒ smaller evening spikes & less peaker burn than the old relationship implies.

③ Difference (period − period), weather‑normalized Meteologica

Input
Two periods' diurnal shapes at the same held weather.
Output
Hour‑by‑hour difference curve (green = midday/other, purple = evening peak).
Read: below zero = less load/burn at the same weather — green midday = solar's bite, purple evening = the battery bite. Quantifies the structural Δ (e.g. −1.5 Bcf/d at 98 °F) to adjust the seasonal NG bias.

📈 Load vs temperature — the raw relationship

ERCOT load vs temperature scatter Meteologica ERA5 poly fit

Input
Cached hourly history (2022–2025, summer + winter): ERCOT load (demand or net load) and temperature (°F). A Zone selector (Whole ERCOT / overlay all 8 / one of the 8 weather zones) — per zone it uses that zone's own observed demand + ERA5 temp. A Resolution selector (Hourly … Yearly) sets the bucket; each point = its mean.
Compute
Least‑squares polynomial fit (Quadratic default, Linear optional) → a smooth regression curve + R². Per‑year or per‑zone curves; WebGL for the dense hourly/overlay views.
Output
Scatter of load vs temperature, coloured by Year / Month / Zone, with the fit curve(s) + an optional P10–P90 quantile band; a per-zone sensitivity map + table beside it (Texas choropleth by cooling GW/°F); a Full-year vs Year-to-date comparison (clip every year to the newest year's window so a partial year compares fairly — +22 % @55 °F 2022→26); a per-zone load time series (TradingView-style, selectable interval, multi-zone tags); and a weather-normalized load-growth curve (each year's demand at normal weather — gap = structural growth, +19 % system-wide / +58 % Far West 2022→25; per-zone, Quadratic or degree-day method). Data: Meteologica observed demand + ERA5 temperature (see data-sources.md §9).
Read: the cloud is a U / hockey‑stick — load up at the cold end (heating) and the hot end (cooling), trough near 65 °F (so the fit is quadratic — linear R²≈0.44 vs 0.94). Colour by Year → the curve shifts up = structural growth (same temp, more load). Overlay the zones → AC‑heavy North Central/Coast have steep cooling arms (~+0.3 GW/°F) while Far West is flat (≈0, industrial) — where heat becomes load. Net load (whole‑ERCOT) flattens it — renewables decouple it. Raw companion to the 🔋 dashboard's quantified drift.
